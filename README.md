## Объяснение простыми словами:

### 1. Проблема бесконечности

Представьте, что врач создает задачу "Ежедневный обход пациентов" и не указывает дату окончания. Если бы мы создавали все экземпляры сразу, нам пришлось бы создать запись на каждый день вперед. Но насколько вперед?

Пример проблемы:
```
Врач создает "Обход" 01.06.2024 без даты окончания.
Если создавать записи на 10 лет вперед:
- Нужно создать 3 650 записей (365 дней × 10 лет)
- Из них реально понадобятся только первые 30-60 записей
- Остальные 3 590 записей просто занимают место в БД
- Если 100 врачей создадут по 5 таких задач = 1 825 000 бессмысленных записей
```

Как решает наше решение:
- Храним только 1 запись-шаблон: "Ежедневный обход, каждый день, начиная с 01.06.2024"
- Когда нужно показать задачи на конкретный месяц (например, июнь) - генерируем 30 дат
- Когда нужно показать на июль - генерируем еще 31 дату
- Никаких записей на годы вперед не создается

Это как хранить не все листы календаря, а только правило: "эта задача есть каждый день".

---

### 2. Независимость состояний

Жизненная ситуация:
```
Задача "Ежедневный обход" на неделю:
Понедельник  - выполнено ✅ (обошел 10 пациентов)
Вторник     - выполнено ✅ (обошел 12 пациентов)
Среда       - отменено ❌ (врач на конференции)
Четверг     - выполнено ✅ (обошел 8 пациентов)
Пятница     - в процессе 🔄 (еще не закончил)
```

Проблема обычного подхода:
Если бы у нас была одна запись на всю серию, то:
- Отметка "выполнено" в понедельник сделала бы выполненной всю серию
- Отмена среды отменила бы всю задачу навсегда
- Нельзя отследить историю по каждому дню

Как решает наше решение:
- У нас есть таблица `task_overrides` - она хранит только те дни, которые чем-то отличаются от "по умолчанию"
- По умолчанию каждый день имеет статус "new" (новая)
- Если врач что-то меняет - создается запись в `task_overrides` для конкретной даты
- Все остальные дни остаются со статусом "new" автоматически

```
Без переопределений (все дни "new"):
Пн: new | Вт: new | Ср: new | Чт: new | Пт: new

С переопределениями:
Пн: done (запись в overrides) | Вт: done (запись) | Ср: cancelled (запись) | Чт: new (нет записи) | Пт: new (нет записи)
```

---

### 3. Минимальное хранение данных

Сколько данных реально хранится:

```
Шаблон задачи "Ежедневный обход":
1 запись в таблице tasks (примерно 100 байт)

За 30 дней работы:
- Создано 30 виртуальных "экземпляров" (ничего не хранится)
- Врач отметил выполненными 25 дней = 25 записей в overrides
- Врач отменил 2 дня = 2 записи в overrides
- 3 дня остались без изменений = 0 записей

Итого: 1 запись шаблона + 27 записей переопределений = 28 записей
Вместо: 30 записей (если бы хранили каждый день)
```

Экономия растет со временем:
```
За 1 год:
- Наше решение: 1 шаблон + ~250 переопределений = 251 запись
- Полное хранение: 365 записей

За 10 лет:
- Наше решение: 1 шаблон + ~2500 переопределений = 2501 запись
- Полное хранение: 3650 записей

Разница: в 1.5 раза для года, но главное - нет проблемы "а что дальше 10 лет?"
```
